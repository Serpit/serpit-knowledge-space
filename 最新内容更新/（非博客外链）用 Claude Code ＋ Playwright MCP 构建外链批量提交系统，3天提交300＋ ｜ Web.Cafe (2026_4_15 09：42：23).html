做独立站 SEO，外链是绕不开的苦活。基本上都是几百个起步，人手发真的要死了，而且我还同时做多个站，肯定得想个别的办法。
我的选择是：用 Claude Code 的 Playwright MCP 直接操控浏览器，一个站一个站地手动提交。听起来笨，但这是我找到的能保证外链质量的方法。
3 天下来，4 个站总计提交了 300+ 条独立域名外链，其中约 40% 是 Dofollow。这篇文章分享整个系统的架构和方法论。

一、系统架构

1.1 核心组件

Claude Code（指挥官）
  ├── Playwright MCP（浏览器操控）
  ├── LinkForge DB（外链管理数据库）
  ├── Skill 知识库（10 个专题文件，~950 行）
  └── Playwright 浏览器 x4（每站独立实例）
Claude Code 负责决策和执行：读取 DB 中的候选站列表，查阅 Skill 知识库中的平台经验，然后通过 Playwright MCP 直接操作浏览器完成注册、填表、提交。
Playwright MCP 是 Anthropic 官方的 @playwright/mcp，提供 browser_navigate / browser_snapshot / browser_click / browser_evaluate 等工具。Claude Code 通过这些工具直接控制 Chrome 浏览器。

1.2 多站并发隔离

4 个站各有独立的：
  • Chrome Profile 目录（cookie/session 完全隔离）
  • ISP 代理端口（每站不同 IP）
  • Playwright MCP 实例（独立浏览器进程）
通过项目级 settings.json 配置，每个 Claude Code session 只启动自己那一个 MCP 实例：
tmux pane 1: cd backlink-site-a && claude   # 启动 site-a 的 MCP
tmux pane 2: cd backlink-site-b && claude   # 启动 site-b 的 MCP
tmux pane 3: cd backlink-site-c && claude   # 启动 site-c 的 MCP
tmux pane 4: cd backlink-site-d && claude   # 启动 site-d 的 MCP
4 个 Claude Code 实例并行跑，互不干扰。

1.3 知识库结构（Skill 文件）

知识库从最初的 2 个巨型文件（3159 行）重组为 10 个职责单一的文件（945 行），缩减 70%：
文件职责行数
SKILL.md入口：流程 + SQL + 索引~100

每次执行外链任务，Claude Code 先读 iron-rules.md（必读），再按需查阅其他文件。新经验追加到对应文件，不堆在一个地方。

二、执行流程

2.1 候选筛选

从 DB 中拉候选站，必须在 SQL 层硬过滤：
-- 核心过滤逻辑（简化版）
WHERE is_spam = 0                    -- 排除 PBN / 链接农场
  AND traffic >= 100                 -- 排除零流量站（DR 是假指标，traffic 是真）
  AND domain NOT IN (已做过的)        -- 按域名去重，不是按模板 ID
ORDER BY traffic DESC                -- 优先按真实流量排序
血泪教训：DR（Domain Rating）是假指标。我们曾经一晚上跑了一批 DR50+ 的候选站，结果全是 PBN 链接农场 -- DR 靠互相刷，实际流量为 0。后来加了 traffic >= 100 硬过滤，一劳永逸。

2.2 单站提交流程

  1. 读铁律（每次都读，防止犯已知错误）
  2. 确认当前站点的品牌信息和锚文本
  3. 查阅目标平台的操作经验
  4. Playwright 打开目标站 → 截图确认页面结构
  5. 按经验操作：注册 / 填表 / 发评论 / 创建 Profile
  6. 提交后实测链接的 rel 属性（nofollow 还是 dofollow）
  7. 写回数据库
  8. Ping 搜索引擎通知新页面
  9. 经验追加到知识库
关键是第 6 步实测 rel。很多平台的 Dofollow 标记是过时的，2026 年大量平台集体加了 nofollow。每次提交后必须跑一段 JS 验证：
document.querySelectorAll('a[href*="你的域名"]')
  .forEach(a => console.log(a.rel || 'EMPTY'))
// EMPTY = Dofollow，含 nofollow = 非 Dofollow

2.3 人机协作模式

大多数操作 Claude Code 自己完成。但有些验证码必须人工：
  1. Claude Code 先填完表单的所有字段
  2. 滚动到验证码区域使其可见
  3. 告诉我"所有字段填完，只剩验证码"
  4. 我手动过验证码
  5. Claude Code 立刻点提交
血泪教训：之前 Claude Code 填了一半字段就让我过验证码，验证码过了但提交时发现漏了 2 个必填字段，浪费了验证码（很多验证码一次性的）。现在铁律规定：必须先填完所有字段，只剩验证码才能叫我。

三、核心技术发现

3.1 六种反垃圾系统的绕过对策

系统能否绕过方法
Akismet能干净 Gmail + ISP 代理 + 正文不放 URL + 链接放 author URL 字段
Antispam Bee能Playwright 原生键盘输入（pressSequentially），不用 JS 设值
WPantispam Protect看配置严格模式：纯文本评论 + 链接放 URL 字段；宽松模式：直接过
CleanTalk不能403 硬拦，看到直接跳过
hCaptcha Enterprise不能评论内容被服务端清洗
Jetpack Highlander不能评论在跨域 iframe 中，无法注入
Akismet 的关键发现：Akismet 判断垃圾评论主要看邮箱信誉和IP 信誉，而不是评论内容。用品牌域名邮箱（team@yourdomain.com）注册过 → 邮箱被标记 → 之后所有评论都被吞。解决方案：用一个干净的 Gmail 地址，名字用随机真实人名，链接只放在 author URL 字段（不放正文）。
Antispam Bee 的关键发现：这个德语圈流行的插件会监听 textarea 的 keydown 事件来确认"真人在打字"。如果用 JavaScript 直接设值（element.value='...'），不触发键盘事件 → 被判定为 bot → 403。解决方案：用 Playwright 的 pressSequentially 逐字符输入，触发完整的键盘事件链。

3.2 前端不行就逆向后端 API

遇到前端操作失败（按钮点不动、表单不提交、Modal 不弹出），第一反应是逆向找后端 API 直接调用。
SOP：
  1. 拉页面所有 inline script（document.querySelectorAll('script')）
  2. 正则提取 axios.post / fetch 的 URL → 得到 API endpoint 清单
  3. 批量试 base URL 前缀（/api/ / /api/v1/ / /ajax/）
  4. 带 session cookie 直接 fetch 调用
真实案例：某 SaaS 目录站的投票按钮点击无反应（Vue mount 失败）。逆向 news.min.js 发现 POST /vote-post endpoint，直接 fetch 调用 5 次 → 15 秒完成原本需要 20 分钟的"访问 5 个产品页面赚积分"流程。
另一个案例：某站注册页的 6 位 OTP 验证码输入框是 6 个独立的 Vue input，自动跳转逻辑在 jQuery onkeyup 里。Playwright 的 fill/type/pressSequentially 都无法正确触发跳转。拦截 XHR 请求发现 POST /signup API，直接带 emailCode 参数调用 → 注册成功。

3.3 2026 年 Dofollow 存活地图

经过 100+ 个平台的实测，2026 年还能拿到 Dofollow 的有以下几类：
第一类：开发者博客平台
  • velog.io（韩国最大开发者博客，GitHub/Google OAuth，Markdown 文章，即发即过）
  • dev.to（GitHub OAuth，文章内链接 Dofollow）
  • telegra.ph（API 创建，DR92，秒做）
  • rentry.co（curl API，秒做）
第二类：SaaS 目录提交
  • 提交产品 URL → 系统自动生成 listing 页 → "Visit website" 按钮默认 Dofollow
  • 识别特征：有 "Submit product/tool" 入口，URL 是品牌短 slug
  • 获取成本极低（只填 1 个 URL），Dofollow 率接近 100%
第三类：论坛 Profile Website 字段
  • phpBB 论坛的 pf_phpbb_website 字段默认 Dofollow（100% 成功率）
  • Boardhost 免费论坛的 Link URL 字段（无需注册，秒发）
  • Discuz 论坛的 Site 字段（注意在 op=info 页面，不是 op=base）
已确认降级的平台（曾经 Dofollow，2026 年加了 nofollow）：
  • paragraph.com、hackmd.io、justpaste.it、codepen.io 等 paste/note 类平台集体降级
  • 规律：平台主动响应 Google 政策变化，给 UGC 内容加 rel="ugc nofollow"

3.4 竞品外链反查的正确姿势

核心发现：同一个站的不同文章，评论审核策略不同。
错误做法：从 Ahrefs 拿到竞品在 example.com 有外链 → 去 example.com 随便找一篇文章发评论 → 被审核/被删。
正确做法：
  1. 从 Ahrefs 拿到竞品的精确 URL（具体到哪篇文章）
  2. 去那篇文章检查竞品评论是否仍然存活
  3. 存活 → 在同一篇文章下发评论（已证明这篇无审核）
  4. 不存活 → 站长开始清理，整站跳过
选文章的 heuristics：
  • 评论数 > 50 且有明显 SEO 评论存活 = 站长不管这篇
  • 老文章（>1 年）评论区更安全 = 站长不再关注
  • 最近有新评论的老文章 = 评论区仍活跃且无审核

四、血泪教训（铁律）

跑了 300+ 条外链，踩了无数坑，最终沉淀出 10 条铁律：
  1. 禁止设限 -- 需要写 800 字 guest post？写。需要注册 + 验证 + 填 20 字段？全填。唯一合法跳过：真付费墙 / 站已死 / CF 硬封
  2. 前端不行先逆向 -- 按钮无反应、Modal 不弹 → 第一反应是找后端 API，不是标跳过
  3. 候选筛选查 spam + traffic -- DR 是假指标，traffic 是真。SQL 层硬过滤
  4. 去重按域名不按模板 ID -- 同一域名可能有多条模板
  5. 查邮件必须开新标签页 -- 绝不 navigate 离开有表单的页面
  6. rel 属性每次实测 -- DB 标记不可信，提交后 JS 验证
  7. 先读知识库再查 DB -- API snippet、隐藏字段名、rel 纠正都在知识库里
  8. 切站必须确认产品 -- 曾经 18 个外链锚文本全写错（A 站的内容写成了 B 站）
  9. catch-all 邮箱失败立刻切 Gmail plus-addressing -- 很多站静默拒绝自定义域名邮箱
  10. 验证码协作：必须先填完所有字段 -- 只剩验证码才叫人

六、关键认知

6.1 外链类型价值排序

SaaS 目录提交 > 开发者博客文章 > 论坛 Profile > WP 评论 > 短链
SaaS 目录是获取成本最低、Dofollow 率最高、存活最持久的外链类型。但 2026 年大量目录站转付费，免费窗口正在关闭。

6.2 关于 Nofollow

很多人觉得 Nofollow 外链没用。实际上：
  • Google 2019 年把 nofollow 从"指令"改为"提示"，现在会选择性传递权重
  • 90%+ Dofollow 比例反而可能被判定为刷 DR，自然分布更安全
  • 高 DR 站的 Nofollow（如 GitHub DR96、Medium DR94）品牌信号价值极高
