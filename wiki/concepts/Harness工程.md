# Harness 工程

## 定义

围绕 [[AI Agent]] 构建的测试、验证与约束基础设施。至少包括四个部分：

1. **验收基线**：什么算做完了
2. **执行边界**：哪些操作不允许
3. **反馈信号**：自动化的对错判断
4. **回退手段**：出错后如何恢复

## 核心判断

**Harness 比模型更关键**——决定系统能不能稳定运行的，往往是外围工程条件。

- 在高可验证任务（代码编写）上最成立
- 在弱验证任务（开放式研究、多轮协商）中，模型上限仍更关键

## 任务象限

| | 目标明确 | 目标模糊 |
|---|---|---|
| **可自动验证** | 最适合 Agent（右上角） | 高效地往错误方向跑 |
| **需人工验证** | 人的审查速度是天花板 | Agent 基本起不到作用 |

Harness 要做的是把任务推进右上角。

## OpenAI 的实践案例
- 约束编码化（Linter/CI）而非文档化
- Agent 端到端自主（从验证到合并）
- 最小化合并阻力（纠错比等待便宜）

## Agentic Engineering 中的三层验证

[[Agentic Engineering工程革命]] 展示了一个具体的 Harness 实践（运营活动配置）：
- **L1**：Go 反序列化校验（结构正确性）
- **L2**：10 项自检清单（5 项 L1 无法检测）
- **L3**：Playwright 前端验证 + API 验证 + 规则引擎测试

AI 流程比人工流程多出 5 层自动化验证，且包含 S10 经验沉淀环节（人工流程完全没有）。
