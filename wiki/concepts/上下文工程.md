# 上下文工程 (Context Engineering)

管理 LLM 输入上下文的系统性方法，决定模型能看到什么、产出什么质量的内容。

## 两个视角

### 工程视角（[[Agent原理架构与工程实践]]）

关注如何**防止 Context Rot**——无关内容占上下文大头导致决策质量下滑。

**分层管理**：
- 常驻层：身份定义、项目约定、绝对禁止项（短/硬/可执行）
- 按需加载：Skills 和领域知识，描述符常驻，完整内容触发时再注入
- 运行时注入：当前时间、渠道 ID 等动态信息
- 记忆层：跨会话经验（MEMORY.md），需要时才读取
- 系统层：Hooks/代码规则处理确定性逻辑，完全不进上下文

**关键原则**：确定性逻辑不放进上下文；稳定的大系统提示比频繁变动的小提示成本更低（Prompt Caching）。

### 认知视角（[[AI正确的废话与Context突破]]）

关注如何**突破 Consensus 天花板**——用个人认知上下文压过训练时的 consensus prior。

**Context Infrastructure 三要素**：
1. **大量积累**：采集客观行为数据（录音、对话、AI 交互历史）
2. **分层提炼**：L1 Observer → L2 Reflector → L3 Axiom，筛选标准是**稳定性**
3. **按需加载**：Skill 系统，每个 skill 是特定任务的 context 子集

**核心判断**：在模型智能跨过一道坎之后，决定产出性质的是 context，不是模型智能。投资个人 context 的收益累积，投资模型智能的收益递减。

## 两个视角的交汇

两者都强调：
- **按需加载**优于**全量注入**
- **分层结构**是管理复杂性的关键
- Skills/Skill 系统是核心实现模式

区别在于：工程视角关注**效率**（token 消耗、缓存命中），认知视角关注**深度**（突破共识、注入判断力）。

### 工程实践视角（[[Agentic Engineering工程革命]]）

关注**上下文稀缺性**作为底层物理约束，以及如何通过工具设计解决。

- **推送模式的失败**：场景路由系统全量推入上下文 → 路由系统本身成为最大噪音源（Context Rot）
- **拉取模式的成功**：平台根据描述自动匹配工具 → 调用时才加载 → 上下文干净
- **[[渐进式披露]]**被多方独立验证为最优解（Anthropic、OpenAI Codex、Cursor、社区）
- **核心公式**：代码输出质量 = AI 能力 × 上下文质量
