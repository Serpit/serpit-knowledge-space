# 你不知道的 Agent：原理、架构与工程实践

**作者**：[[@HiTw93]]
**发布日期**：2026-03-19
**来源**：X (Twitter)

## 核心观点

一篇系统梳理 AI Agent 工程实践的长文，覆盖控制流、[[上下文工程]]、工具设计、记忆系统、多 Agent 组织、评测、追踪和安全。

**关键判断**：更贵的模型带来的提升，很多时候没有想象中大；[[Harness工程]]和验证测试质量对成功率影响更大；调试 Agent 应优先检查工具定义而非怀疑模型能力。

## 要点摘要

### 1. Agent Loop
- 核心循环只有 ~20 行代码：感知 → 决策 → 行动 → 反馈
- 新能力通过三种方式接入：扩展工具集、调整系统提示结构、状态外化到文件/数据库
- 不应让循环体变成巨大的状态机

### 2. Harness 比模型更关键
- Harness = 验收基线 + 执行边界 + 反馈信号 + 回退手段
- OpenAI 实践：3 个工程师 5 个月写了百万行代码（10x 速度），关键在于约束编码化、Agent 端到端自主、最小化合并阻力
- 任务清晰度 × 验证自动化 = Agent 发挥空间

### 3. [[上下文工程]]
- Context Rot：无关内容占上下文大头，决策质量下滑
- 分层管理：常驻层（短/硬/可执行）→ 按需加载（Skills）→ 运行时注入 → 记忆层 → 系统层
- Prompt Caching：稳定前缀命中率高，大系统提示反而成本更低
- Skills 按需加载：描述符要短且像路由条件，反例使准确率从 53% 升到 85%

### 4. 工具设计（ACI 原则）
- 三代演进：API 封装 → ACI（面向 Agent 目标）→ Advanced Tool Use（动态发现 + 代码编排 + 示例驱动）
- 结构化错误 + 修正建议 > 简单错误字符串
- 5 个 MCP 服务器 ≈ 55,000 tokens 工具定义开销

### 5. 记忆系统
- 四种记忆：工作记忆（上下文窗口）、程序性（Skills）、情景（JSONL 历史）、语义（MEMORY.md）
- 与 [[Mem0]] 的区别：不止蒸馏到事实层，要深入到判断原则层
- 整合流程必须可回退：只移动指针，不删除原始消息

### 6. 多 Agent 组织
- 指挥者模式（同步）vs 统筹者模式（异步委派）
- 子 Agent 只回传摘要，探索细节留在自己的上下文
- 幻觉会互相放大 → 需要交叉验证打断错误链

### 7. 评测
- Pass@k（能力边界）vs Pass^k（回归质量），不能混用
- 先修评测，再改 Agent — 评测系统自身 bug 常伪装成模型退化
- 从 20-50 个真实失败案例启动

### 8. [[OpenClaw]] 实现
- 五层解耦架构：Gateway → Channel 适配器 → Pi Agent → 工具集 → 上下文+记忆
- SOUL.md 定义身份/约束/完成标准
- cron + heartbeat 主动触发模式
- 安全：白名单授权 + 工作空间隔离 + 操作审计日志

## 值得深入的问题
- Harness 工程如何在非代码任务（如研究、写作）中适配？
- Skills 路由准确率在工具数 >50 时的衰减情况
- 多 Agent 幻觉放大的量化评估方法
