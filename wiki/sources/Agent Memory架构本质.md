# Agent Memory 架构本质

**作者**：[[lencx]]（公众号"浮之静"）
**来源**：[mp.weixin.qq.com](https://mp.weixin.qq.com/s/aBWunszbTVjI4bdguLIOvg)

## 质量评估

`overall: 5 / opinion`

信息密度与原创性均为 5：每段都是独立论点，几乎无废话；明确与"记忆 = 蒸馏"划界，给出六维记忆单元、五类型、三链路闭环的自成体系工程化切分。结构清晰度 5（瓶颈 → 边界 → 蒸馏 → 建模对象 → 单元 → 链路 → 进化 → 落地），术语都有定义。事实性 4，benchmark 级声明（35 session/300 turn、selective forgetting 已被单独考核）未给出具体出处，但未发现编造。作者自称"暴论"风格，立场标签为 `opinion`，非客观综述。

## 核心观点

**Memory 的难点从来不在容量，在治理**。Context window 扩展（128K/200K/1M）解决的是带宽问题，不是建模问题；记忆需要一个完整的 **write–manage–read** 闭环，而非"有个存储层就算有记忆"。

## 瓶颈在持续理解

- 模型在单次会话里已足够聪明，问题是"没法把昨天学到的东西以可靠、可更新、可追责的方式带到今天"。
- 一个数据分析 Agent 的三层理解（工具偏好 → 业务焦点 → 沟通语境）需要不同时间尺度才能浮现，没有记忆系统只能永远做"第 1 次会话的事"。
- Benchmark 证实：拉到 35 个 session、300 个 turn 尺度上，长上下文与 RAG 在时间推理、长程一致性上仍明显落后于人类。

## 概念边界

严格区分 Memory 与四个邻居概念：

- **Memory ≠ State**：State 是 session 内短期运行态（对话上下文、工具中间结果、规划器当前步骤），结束即销毁。Memory 是跨 session 持续存在的结构化历史。
- **Memory ≠ Policy**：Policy 管"允许与禁止"（权限、安全、合规），是外部规范，不应被 memory 动态修改。**如果 Agent 的记忆能改写自己的权限规则，那不是记忆在进化，是越权。**
- **Memory ⊃ Profile**：Profile 是用户模型的低维、显式、便于消费的快照层（名字、角色、偏好标签），是记忆的输出产物，不是记忆本身。
- **Memory ≠ 蒸馏**：摘要/reflection/session summary 是 memory pipeline 的管理动作，不是 memory 本身。

一句话定义：**Memory 保存的是可跨时延续并影响未来决策的结构化历史——"结构化"指带来源、作用域、时间权重和可修正性的历史对象，而不是"把聊天记录再存一份"**。

## 蒸馏的位置与局限

- 蒸馏类似"通信系统里的压缩"——是管道的一步，不是管道本身。
- 局限：天然偏向**静态结论**。能写下"用户偏好 TypeScript"，却很难保留这条偏好**如何形成**、**在什么上下文下成立**、**最近是否在漂移**。
- 蒸馏擅长留下结论，不擅长留下**形成结论的轨迹**。对生产级 Agent 来说，**轨迹往往比结论更值钱**。
- 系统做完摘要就停了、没有后续的冲突检测、信念衰减、回溯修正 → 这不是记忆，只是归档。

## 四个建模对象

一种高价值的切分方式（非唯一分类法）：

| 对象 | 内容 | 典型信号 |
|---|---|---|
| **用户模型** | 偏好、风险偏好、沟通习惯、决策模式 | "抵触 TS → 逐渐接受 → 主动要求重写"的转变轨迹 |
| **任务模型** | 方案否决、结论确认、真版本 artifact、未完成承诺 | 很多 Agent 失败不是不懂你，而是不记得事情推进到哪一步 |
| **世界模型** | 仓库结构、API 约束、系统边界、组织规则、数据新鲜度 | 大量"个性化错误"本质是没记住环境变了 |
| **自我模型** | 试过什么、哪条路径失败过、哪个工具何时不稳定、哪些推断是暂定假设 | 没有这层，Agent 不是在学习，只是在重复犯错 |

**意图不是单独字段**，而是四层长期耦合后浮现的上层能力——像跟了你三年的助理，"懂你"不是因为背偏好手册，而是同时理解脾气、项目进度、组织环境和他自己的能力边界。

## 记忆单元六维

- **内容**：这条记忆说了什么。
- **类型**：至少五种——
  - `event`（发生了什么）
  - `assertion`（用户明确声明的，可被用户直接推翻）
  - `belief`（Agent 推断出来的，需新证据逐步修正）
  - `constraint`（不可违反的边界，权威来源在 memory 子系统外部，memory 仅记录和引用）
  - `commitment`（Agent 做出但尚未完成的承诺）
- **置信度**：主要适用 belief / commitment。对 event 和 assertion，"发生"的确定性 ≠ "含义是否仍成立"的确定性，二者不可混。
- **来源（provenance）**：用户明说 / 行为推断 / 环境观察 / Agent 自生成。决定可信层级与可撤销性。**没有 provenance，Agent 无法区分一条扎实的用户声明和一次自己的高置信幻觉**。
- **作用域（scope）**：在什么上下文下成立。"偏好性能优先"在后端架构决策中成立，在前端原型阶段未必。没有 scope，信念会被过度泛化。
- **时间与衰减**：产生时间、上次确认/引用时间、衰减权重。

六维钉住 → 记忆从"一堆字符串"变成可查询、可追溯、可修正、可过期的结构化对象。

## 三条链路闭环

### 写入：decision under budget

- 预算包含：存储空间、未来检索成本、推理注意力开销、冲突管理代价。
- 写入要看**相对于已有记忆的边际价值**：第四次观察到同样信号的边际价值远低于第一次。
- **冲突信号本身是高价值信号**：一个一直保守选型的用户突然要求尝试 alpha 框架，这个偏移值得优先写入。
- **行为证据 > 口头表态**：用户说"我不喜欢 ORM"是 assertion；连续三次在 ORM 方案后又手写 SQL 是 belief，后者 provenance 更硬。

### 管理：最容易偷懒也最关键

五件事决定了"资产 vs 垃圾堆"：

1. **整合**：碎片信号 → 结构化信念，蒸馏是手段之一。
2. **冲突处理**：不要用"以最新为准"（偷懒的蒸馏思维），而是保留矛盾、建模为"此维度偏好是情境依赖的"，读取时按当前情境选择。
3. **衰减与遗忘**：用户去年偏好 REST 不代表今年没迁移；遗忘不是 bug，是防止过拟合现实的必要机制。
4. **来源追踪**：没有 provenance 就无法判断信念可信度，也无法出错时回溯责任链。
5. **权限治理**：用户必须能查看/编辑/删除 Agent 的记忆——不只是合规，更是信任基础。

### 读取：任务约束优先

- 传统 RAG 假设"相关性由表面语义决定"，在记忆场景下有根本局限。
- **反直觉案例**：用户问"帮我写缓存方案"，最相关的记忆可能是三个月前提到的"黑五流量问题"——决定了设计约束，但在语义空间里与"缓存"距离很远。
- 从 `retrieve(query)` 升级到 `read(task_context, belief_graph)`：先由任务理解层判断"当前决策真正受什么约束"，再由检索层找对应记忆，最后评估这些记忆在当前情境下的适用性。

## 进化 = 修正 + 遗忘

- **自我修正**：负反馈应回溯到记忆层——是召回错了？belief 过期了？belief 没错但被错误应用到了当前 scope？**只在回答层修补而从不修正上游假设，不是学习，是打补丁**。
- **策略性遗忘**：benchmark 已单独考核 selective forgetting、knowledge update、preference drift。该忘的：被后续信号反复否定的旧 belief、高度情境依赖且低泛化的细节、已被更高层抽象吸收的底层 event。
- 金句：**死的不是经验本身，而是那些失去了更新机制的经验**。Few-shot 示例、摘要、fine-tuned preference profile 并不天然低级——问题是一旦脱离持续校正闭环，就从资产变成惯性。

## 落地判断

- 不谈数字分身的宏大叙事，成熟记忆系统逼近的不是"复制你"，而是**在特定任务域中近似你的判断风格与约束偏好**。
- 四个核心动词都指向治理：写入决定什么信息**获得**对未来的影响力；管理决定什么信念**继续**有效；读取决定什么记忆**真正进入**当下决策；遗忘决定什么经验**退出**舞台。
- 评测转向：从"能不能 recall"到"能不能 update / abstain / handle drift / selective forget"。

## 与已有资料的连接

- 与 [[Mem0]] 的批评同向：Mem0 蒸馏到事实层就停了，而本文指出**只做摘要不做冲突检测、信念衰减、回溯修正就是归档**；两个批评从不同角度指向同一问题。
- 与 [[鸭哥]] 的 [[上下文工程|Context Infrastructure]] 互补：鸭哥关注**认知层的判断原则蒸馏**（突破 Consensus 天花板），lencx 关注**记忆层的治理闭环**（跨 session 可持续影响决策）。前者是"写什么进上下文"，后者是"怎么管跨时的结构化历史"。
- 与 [[Harness工程]] 的交汇：权限治理、provenance、scope 本质都属于 Agent 的外围约束基础设施。

## 值得深入的问题

- 置信度、作用域、衰减权重如何量化计算？是否需要领域先验？
- "情境依赖冲突建模"的 schema 长什么样？情境怎么打标、读取时怎么匹配？
- `read(task_context, belief_graph)` 如何与现有 RAG 栈（向量库、BM25、rerank）耦合？task_context 的结构化表示是什么？
- `constraint` 类型记忆的权威来源在 memory 子系统之外——这个外部 registry 和 policy 系统是否应当统一？
- 五类型（event / assertion / belief / constraint / commitment）的存储后端是否应当分离？更新语义差异能否在同一套检索接口下被抽象？
