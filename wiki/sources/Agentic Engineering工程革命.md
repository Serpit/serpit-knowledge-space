# 认知重建之后，步入Agentic Engineering的工程革命

**作者**：rickyshou
**来源**：系列文章第三篇

## 概述

从一个 AGENTS.md 文件出发，两个月内自然生长为拥有 22 个 Agent、27 个 Skill、28 个命令的工程体系。这是一份带着所有弯路和修正的实践记录，覆盖了从概念到团队落地的全过程。

## 核心观点

### Agentic Engineering 的定义

人从"代码编写者"转变为"Agent 协调者"，通过系统化的[[上下文工程]]、知识沉淀和流程设计，让 [[AI Agent]] 具备自主性、记忆力和成长性，在不牺牲质量的前提下实现工程效率的复利增长。

不是 Vibe Coding 的对立面，而是其**工程化演进**——保留杠杆效应，补上质量保障和知识复利。

### 关键认知突破

1. **Agent 的"接口"不是函数签名，而是上下文窗口里的信息组合**。用 DDD 的确定性边界无法约束概率性的 Agent 行为。
2. **上下文空间是稀缺资源**。任何"管理机制"都消耗上下文，管理机制的收益必须大于它消耗的认知带宽。
3. **位置即语义**——文件放在哪个目录，AI 就知道它是什么。

## 演进过程（4 个阶段）

### 阶段一：一个文件起步（1月初）
- 一个 AGENTS.md 定义角色和规则
- 立刻遇到：无记忆、经验不沉淀、状态不持久化

### 阶段二：目录结构涌现（1月上旬）
- `context/` 自然分化为团队级（team/）和项目级（project/）
- `requirements/{id}/` 涌现固定结构：meta.yaml、plan.md、process.txt、notes.md
- 三层记忆系统：长期记忆（context/+skills/+requirements/）→ 溢出区（process.txt/notes.md）→ 工作记忆（对话上下文）

### 阶段三：场景驱动架构（1月中旬）
- 受 Anthropic "最小约束"指导，引入场景路由机制
- 早期运行良好，但随场景增长出现 **Context Rot**
- 路由系统本身成为最大的上下文噪音源

### 阶段四：推翻场景路由，转向工具设计（1月下旬）
- **关键转折**：场景路由（推送模式）→ 工具设计（拉取模式）
- 推送：路由 Agent 检索 → 全量推入上下文 → 噪音越来越大
- 拉取：平台根据描述自动匹配工具 → 调用时才加载 → 上下文干净
- 工具设计把"理解意图"和"加载规范"解耦

## 方法论体系

### 三级能力体系（按上下文影响分层）

| 类型 | 上下文影响 | 适用场景 |
|---|---|---|
| Command（<100行） | 极小 | 意图快捷入口，预检+委托 |
| Skill（<2k token） | 可控 | 专业知识包，渐进式披露 |
| Subagent | 隔离（仅摘要返回） | 产生大量输出或需独立探索 |

### 工具设计哲学
- **封装知识而非流程**——告诉 Agent "做什么"和"注意什么"，不是"第一步第二步"
- Skill 渐进式披露：SKILL.md <2k token 进入上下文，详细内容拆到 resources/
- Subagent 硬约束：返回 <2k token，禁止嵌套调用

### SOP 迭代决策树
- 六种触发场景 → 定位问题层级 → 确定变更层（设计指导/功能预期/场景规范/工具）→ 判断是否工具化 → 迭代检查清单

## 实战案例：运营活动配置

### Ralph Loop 机制
- 把同一 Prompt 反复喂给 AI，状态通过文件系统传递
- Agent 每轮读取 state.json，按 current_phase 执行对应步骤
- 完成时输出 `<promise>` 标记，Stop Hook 检测后允许退出
- 适合多步骤、跨系统、需迭代修复的任务

### 10 步执行流程
S1 需求解析 → S2 商品创建 → S3 活动创建 → S4 配置生成 → S5 Go 校验 → S6 DB 写入 → S7 API 验证 → S8 前端验证（Playwright）→ S9 玩法引擎测试 → S10 经验沉淀

- 原来 3-5 人天/活动，目标缩短到分钟级
- AI 流程比人工流程多出 5 层自动化验证（S5-S9）
- S10 经验沉淀是人工流程完全没有的

### 三层验证
- L1：Go 反序列化校验（结构正确性）
- L2：10 项自检清单（5 项 L1 无法检测）
- L3：Playwright 前端验证 + API 验证 + 规则引擎测试

## 多方收敛验证

五个独立实践收敛到同一组设计原则：
1. **Anthropic**：Agentic Search 理论框架
2. **OpenAI Codex 团队**：progressive disclosure，百万行代码验证
3. **everything-claude-code**：社区十个月独立演化
4. **Cursor**：动态上下文发现，token 消耗减少 46.9%
5. **作者**：全生命周期工程体系

共识：上下文稀缺性是底层物理约束，渐进式披露是最优解，机械化强制执行是可持续的必要条件。

## 团队落地四阶段

| 阶段 | 策略 | 衡量标准 |
|---|---|---|
| 种子期 | 结果先行 + 知识库兜底 | 有同事自行走完前三阶段 |
| 扩散期 | clone 替代培训，隐藏复杂度 | 经验共享循环转起来 |
| 加速期 | 嵌入式推进（参考 Anthropic 驻场高盛） | 嵌入结束时项目有第一批业务知识 |
| 常态期 | 制度化非强制化 | 人员离开时知识仍在 context/ 中 |

### 团队认知转变
1. 从"AI 是工具"到"AI 是团队成员"
2. 从"个人技巧"到"团队知识"
3. 从"一次性使用"到"持续投资"

**核心公式**：代码输出质量 = AI 能力 × 上下文质量

## 做对了什么
- 坚持"先跑起来"，每个设计决策都有失败案例作论据
- 让具体问题驱动演进，过滤掉过度设计冲动
- 知识沉淀是工作一部分，不是事后附加
- "文档即记忆"：Markdown + git，人和 AI 读同一份文件

## 踩过的坑
- 规范膨胀：每次 Agent 犯错就加规则 → 上下文过载 → Agent 走神
- 修正：定期"规范瘦身"，偶发性错误不维护永久约束

## 值得深入的问题
- Ralph Loop 在非编码任务中的适用性
- 22 个 Agent 的具体分工和模型分级策略
- context/ 知识库的腐化检测和自动清理机制
