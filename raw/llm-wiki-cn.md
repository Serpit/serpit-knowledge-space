# LLM Wiki

一种使用 LLM 构建个人知识库的模式。

这是一个创意文档，旨在被复制粘贴到你自己的 LLM Agent 中（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达高层理念，而具体细节将由你的 Agent 与你协作构建。

## 核心理念

大多数人使用 LLM 处理文档的体验类似于 RAG：你上传一组文件，LLM 在查询时检索相关片段，然后生成答案。这种方式可行，但 LLM 每次都在从零开始重新发现知识。没有积累。问一个需要综合五份文档才能回答的微妙问题，LLM 每次都得重新找到并拼凑相关片段。什么都没有被构建起来。NotebookLM、ChatGPT 文件上传，以及大多数 RAG 系统都是这样工作的。

这里的理念不同。LLM 不是在查询时才从原始文档中检索，而是**逐步构建并维护一个持久化的 wiki**——一个结构化的、相互链接的 Markdown 文件集合，它位于你和原始资料之间。当你添加新资料时，LLM 不只是为以后检索而索引它。它会阅读资料、提取关键信息，并将其整合到现有 wiki 中——更新实体页面、修订主题摘要、标注新数据与旧结论相矛盾之处、强化或质疑不断演进的综合分析。知识被编译一次然后_持续更新_，而不是每次查询都重新推导。

这就是关键区别：**wiki 是一个持久的、不断增值的产物。** 交叉引用已经存在。矛盾已经被标记。综合分析已经反映了你读过的所有内容。每添加一个新资料、每问一个新问题，wiki 就变得更丰富。

你从不（或很少）亲自编写 wiki——LLM 编写并维护所有内容。你负责资料搜集、探索和提出正确的问题。LLM 做所有繁重的工作——总结、交叉引用、归档和记账，这些才是让知识库随时间推移真正有用的关键。在实践中，我一边打开 LLM Agent，一边打开 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果——跟踪链接、查看图谱视图、阅读更新的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这可以应用于很多不同的场景。举几个例子：

- **个人**：跟踪你自己的目标、健康、心理、自我提升——归档日记条目、文章、播客笔记，随时间建立一个关于你自己的结构化画像。
- **研究**：在数周或数月内深入一个主题——阅读论文、文章、报告，逐步构建一个包含不断演进的论点的综合性 wiki。
- **读书**：逐章归档，为人物、主题、情节线索建立页面，以及它们之间的联系。读完后你会得到一个丰富的伴读 wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki——数千个互相链接的页面涵盖人物、地点、事件、语言，由志愿者社区历经多年建成。你可以在阅读时借助 LLM 个人构建类似的东西，让 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，输入来源为 Slack 对话、会议记录、项目文档、客户电话。可能需要人工审核更新。wiki 保持更新是因为 LLM 做了团队中没人愿意做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、兴趣深入研究**——任何你在持续积累知识并希望将其组织起来而非散落各处的场景。

## 架构

有三个层次：

**原始资料** — 你精心收集的源文档。文章、论文、图片、数据文件。这些是不可变的——LLM 从中读取但从不修改。这是你的事实来源。

**Wiki** — 一个由 LLM 生成的 Markdown 文件目录。摘要、实体页面、概念页面、对比、概览、综合分析。LLM 完全拥有这一层。它创建页面，在新资料到达时更新它们，维护交叉引用，并保持所有内容一致。你阅读它；LLM 编写它。

**Schema（模式）** — 一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM wiki 的结构、约定，以及在摄入资料、回答问题或维护 wiki 时应遵循的工作流程。这是关键的配置文件——它使 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用聊天机器人。你和 LLM 会随着时间推移共同演进这个文档，逐步弄清什么对你的领域最有效。

## 操作

**摄入（Ingest）。** 你把一个新资料放入原始集合并告诉 LLM 处理它。一个示例流程：LLM 读取资料，与你讨论关键要点，在 wiki 中写一个摘要页面，更新索引，更新 wiki 中相关的实体和概念页面，并在日志中追加一条记录。一个资料可能涉及 10-15 个 wiki 页面。就我个人而言，我倾向于逐个摄入资料并保持参与——我阅读摘要、检查更新，并引导 LLM 关注重点。但你也可以批量摄入多个资料，减少监督。开发适合你风格的工作流程并将其记录在 schema 中供未来会话使用，这取决于你。

**查询（Query）。** 你针对 wiki 提出问题。LLM 搜索相关页面，阅读它们，并综合出带有引用的答案。答案可以根据问题类型采取不同形式——一个 Markdown 页面、一个对比表、一个幻灯片（Marp）、一个图表（matplotlib）、一个画布。重要的洞察：**好的答案可以作为新页面归档回 wiki。** 你要求的对比、分析、发现的联系——这些很有价值，不应该消失在聊天历史中。这样你的探索就像摄入的资料一样，在知识库中不断积累。

**审查（Lint）。** 定期让 LLM 对 wiki 进行健康检查。查找：页面之间的矛盾、被更新资料取代的过时声明、没有入站链接的孤立页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 擅长建议新的研究问题和新的资料来源。这让 wiki 在增长过程中保持健康。

## 索引和日志

两个特殊文件帮助 LLM（和你）在 wiki 增长时进行导航。它们服务于不同的目的：

**index.md** 是面向内容的。它是 wiki 中所有内容的目录——每个页面列出链接、一行摘要，以及可选的元数据如日期或资料数量。按类别组织（实体、概念、资料等）。LLM 在每次摄入时更新它。回答查询时，LLM 先读取索引找到相关页面，然后深入阅读。这在中等规模（约 100 个资料，数百个页面）下出奇地好用，避免了对基于嵌入的 RAG 基础设施的需求。

**log.md** 是按时间顺序的。它是一个只追加的记录，记载了什么时候发生了什么——摄入、查询、审查。一个有用的技巧：如果每条记录以一致的前缀开头（例如 `## [2026-04-02] ingest | 文章标题`），日志就可以用简单的 Unix 工具解析——`grep "^## \[" log.md | tail -5` 可以获取最后 5 条记录。日志为你提供 wiki 演变的时间线，并帮助 LLM 了解最近做了什么。

## 可选：CLI 工具

到一定程度后，你可能想构建一些小工具来帮助 LLM 更高效地操作 wiki。wiki 页面的搜索引擎是最明显的——在小规模下，索引文件就够了，但随着 wiki 增长，你需要适当的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个本地 Markdown 文件搜索引擎，支持 BM25/向量混合搜索和 LLM 重排序，全部在本地运行。它既有 CLI（LLM 可以调用它）也有 MCP 服务器（LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的东西——随着需求出现，LLM 可以帮你快速编写一个简单的搜索脚本。

## 技巧和窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，可以将网页文章转换为 Markdown。非常适合快速将资料收入你的原始集合。
- **本地下载图片。** 在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设为固定目录（例如 `raw/assets/`）。然后在设置 → 快捷键中，搜索"Download"找到"下载当前文件的附件"并绑定快捷键（例如 Ctrl+Shift+D）。剪裁文章后，按快捷键即可将所有图片下载到本地。这是可选的但很有用——它让 LLM 可以直接查看和引用图片，而不依赖可能失效的 URL。注意 LLM 不能在一次读取中原生处理带有内联图片的 Markdown——变通方法是让 LLM 先读取文本，然后单独查看部分或全部引用的图片以获取额外上下文。有点笨拙但效果还不错。
- **Obsidian 的图谱视图**是查看你的 wiki 形态的最佳方式——什么与什么相连，哪些页面是枢纽，哪些是孤岛。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有相应插件。适合直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以对页面 frontmatter 运行查询。如果你的 LLM 为 wiki 页面添加了 YAML frontmatter（标签、日期、资料数量），Dataview 可以生成动态表格和列表。
- Wiki 就是一个 Markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作功能。

## 为什么这行得通

维护知识库中令人厌烦的部分不是阅读或思考——而是记账。更新交叉引用、保持摘要更新、标注新数据与旧声明的矛盾、在数十个页面间保持一致性。人类放弃 wiki 是因为维护负担的增长速度超过了价值。LLM 不会厌倦、不会忘记更新交叉引用，而且可以一次操作 15 个文件。wiki 保持维护状态是因为维护成本接近于零。

人类的工作是筛选资料、指导分析、提出好问题，并思考这一切意味着什么。LLM 的工作是其他一切。

这个理念在精神上与 Vannevar Bush 的 Memex（1945年）相关——一个个人的、精心策划的知识存储，文档之间有关联路径。Bush 的愿景比 Web 实际发展的样子更接近于此：私有的、积极策划的，文档之间的联系与文档本身一样有价值。他无法解决的部分是谁来做维护。LLM 解决了这个问题。

## 说明

本文档故意保持抽象。它描述的是理念，而非具体实现。确切的目录结构、schema 约定、页面格式、工具——所有这些都取决于你的领域、你的偏好和你选择的 LLM。上面提到的一切都是可选的和模块化的——选择有用的，忽略不需要的。例如：你的资料可能只是纯文本，所以根本不需要图片处理。你的 wiki 可能足够小，索引文件就够了，不需要搜索引擎。你可能不关心幻灯片只想要 Markdown 页面。你可能想要完全不同的输出格式。正确的使用方式是将它分享给你的 LLM Agent，一起协作来实例化一个适合你需求的版本。本文档的唯一使命是传达这个模式。你的 LLM 会搞定剩下的。
